# 10-專案初始化工作模式

> 2026-05-24 更新：本文件已改為自含式 Skill 安裝文件。請使用文末「內建 Skill 完整安裝內容」，不需要額外的舊版獨立 skills 子目錄。


## 目標

把「開始新專案」變成固定工作模式，不只套用在單一專案。

## 前置條件

- 已完成 `README.md` 的設定表。
- 已決定 `{{WORK_ROOT}}`。
- 若使用 Obsidian，已完成 `04-建立第二大腦-Obsidian`。
- 若使用 GitHub，已完成 `03-連接-GitHub`。
- 若使用個人助手資料層，已完成 `09-個人助手設定`。

## 全域規則位置

Codex 全域規則：

```text
{{SYNC_ROOT}}/core-rules.md
```

`{{CODEX_HOME}}/AGENTS.md` 只作為 Codex App 的 symlink 入口，應指向 `{{SYNC_ROOT}}/core-rules.md`。其他 AI agent 要共用同一套規則時，直接讀 `{{SYNC_ROOT}}/core-rules.md`；不要另建 `{{SYNC_ROOT}}/agents/AGENTS.md`。

建議加入：

```markdown
## 新專案固定工作模式

當我說「開始新專案」、「新專案初始化」、「初始化專案工作模式」、「建立專案」時：

- 先檢查環境與現有資料夾，不一開始就改檔。
- 先問清楚專案名稱、用途、工作資料夾、GitHub repo、公開或私有、是否啟用 GitHub Pages、Obsidian 駕駛艙位置、是否使用 Firebase、是否使用 Netlify、部署目標。
- 工作資料夾預設放在 `{{WORK_ROOT}}`。
- Obsidian 只放專案駕駛艙：`專案庫/<專案名稱>/專案工作流程.md`。
- 專案根目錄要有 `AGENTS.md`、`README.md`、`.gitignore`、Git repo。
- 若使用個人助手資料層，專案需要本地 assistant skill 或本地記憶時，建立 `000_Agent/skills/` 與 `000_Agent/memories/`；全域 Codex skills 才放 `{{CODEX_HOME}}/skills`。
- 既有專案只補缺口，不覆蓋既有設定或 Git 歷史。
- 不把 secrets 寫進 repo。
```

放在：

```text
{{CODEX_HOME}}/skills
```

建議建立：

- `project-init-sync`

## 新專案標準流程

1. 先檢查 `{{WORK_ROOT}}` 與是否已有同名資料夾。
2. 問清楚：
   - 專案名稱
   - 用途
   - 工作資料夾
   - GitHub repo 是否需要
   - public / private
   - GitHub Pages 是否需要
   - Obsidian 駕駛艙路徑
   - Firebase 是否需要
   - Netlify 是否需要
   - 部署目標
3. 建立工作資料夾：

```text
{{WORK_ROOT}}/<project-name>
```

4. 建立或補齊：
   - `AGENTS.md`
   - `README.md`
   - `.gitignore`
   - Git repo
5. 建立 Obsidian 駕駛艙：

```text
{{OBSIDIAN_PROJECTS}}/<project-name>/專案工作流程.md
```

6. 若使用個人助手資料層，建立專案本地：

```text
100_Todo/
  drafts/
  projects/
  archive/
200_Reference/
  writing-samples/
  templates/
  past-work/
```

只有需要專案本地 assistant skill 或專案獨立記憶時，才建立：

```text
000_Agent/skills/
000_Agent/memories/
```

`000_Agent/skills` 只放該專案自己的 assistant skill、流程草稿或局部工具說明；不要 symlink 到 `{{CODEX_HOME}}/skills`。需要全域觸發的 Codex skill 才安裝到 `{{CODEX_HOME}}/skills`，並依 `16-Codex-全域-Skills-跨裝置同步` 指向 `{{SYNC_ROOT}}/skills`。
`000_Agent/memories` 只在專案真的需要獨立助手記憶時才建立。

7. 若需 GitHub，再建立 repo 與 remote。
8. 若需 Firebase，再建立或連接 project。
9. 若需 Netlify，先確認 Netlify CLI/MCP 可用、team、site name 或 site ID、output folder、production/preview 意圖，再建立或連結 site。

## 專案 AGENTS.md 範本

```markdown
# <project-name> — AGENTS.md

## 專案入口

- 專案名稱：<project-name>
- 專案用途：<purpose>
- 工作目錄：`{{WORK_ROOT}}/<project-name>`
- GitHub repo：`{{GITHUB_USER}}/<repo>` 或 `未建立`
- Obsidian vault：`{{OBSIDIAN_VAULT}}`
- 專案駕駛艙：`專案庫/<project-name>/專案工作流程.md`
- GitHub Pages：`已啟用 / 未啟用`
- Firebase project：`{{FIREBASE_PROJECT_ID}}` 或 `未使用`
- Netlify：`{{NETLIFY_SITE_NAME_OR_ID}}` 或 `未使用`
- 部署目標：`{{DEPLOYMENT_TARGET}}` 或 `暫無`

## 開工

- 讀本檔。
- 讀 Obsidian 駕駛艙。
- 檢查 Git 狀態。
- 不自動 pull、commit、push。

## 收工

- 更新 Obsidian 駕駛艙。
- 回報 Git 狀態。
- 只有使用者要求時才 commit / push。

## 安全

- 不提交 `.env`、API key、token、密碼、Admin 憑證、個資或敏感資料。
```

## 既有專案規則

若專案已存在：

- 先盤點已存在 / 缺少項目。
- 只補缺口。
- 不覆蓋既有 `AGENTS.md`、`README.md`、`.gitignore`、Firebase 設定或 Git 歷史。
- 只補上個人助手整合段落與缺少的本地 `100_Todo/`、`200_Reference/`；若該專案有本地 assistant skill 或記憶需求，再補 `000_Agent/skills/`、`000_Agent/memories/`。

## 命名規則

新專案資料夾建議使用：

- 小寫英文
- 數字
- 底線 `_`
- 連字號 `-`

避免空格。

## 驗證

1. 專案資料夾存在。
2. `AGENTS.md`、`README.md`、`.gitignore` 存在。
3. `git status --short --branch` 可執行。
4. Obsidian 駕駛艙存在。
5. 若有 GitHub，`git remote -v` 正確。
6. 若有 Firebase，`.firebaserc` 指向正確 project。
7. 若有 Netlify，確認 `.netlify/`、`netlify.toml` 或 Netlify CLI/MCP 狀態與 Obsidian 駕駛艙一致。

## 設定範例

本機曾使用專案名稱 `codex_installation`。這只是範例；下載者請使用自己的專案名稱。

## 踩坑修正

- 新增全域 skills 後，要重開 Codex 對話或重啟 App。
- 來源文件若提到 來源工具規則檔 或 來源工具的 skills 路徑，要先判斷層級：全域觸發才轉成 `{{CODEX_HOME}}/skills`；專案/助手本地 skill 轉成 `000_Agent/skills`。
- Codex 側邊欄不會即時跟著 Finder / 雲端硬碟改名刷新。
- 專案改名時要同步改：
  - 工作資料夾
  - `{{CODEX_CONFIG}}` 信任路徑或 MCP 路徑
  - 專案 `AGENTS.md`
  - Obsidian 專案庫資料夾
  - Obsidian 駕駛艙內文
  - Firebase MCP project directory
  - Netlify site name / site ID / output folder 記錄
- GitHub 若沒有 remote，就不需要改 repo。
- Firebase Project ID 不會因本地專案改名而改變。
- Netlify Site ID 不會因本地專案改名而自動改變。

## 內建 Skill 完整安裝內容

使用方式：把下方整段安裝腳本複製到自己的環境執行。執行前請先把 `{{CODEX_HOME}}` 替換成自己的 Codex 設定資料夾，例如 `{{CODEX_HOME}}`。

set -e

Follow lazy pack #10 as the default standard for all future new projects: inspect first, ask for missing project facts, create AGENTS.md, README, .gitignore, Git repo, optional GitHub repo, optional Firebase link, optional Netlify link, and an Obsidian project cockpit without overwriting existing work.

# Project Init Sync

Use this workflow for every new project unless the user explicitly says not to.

## Default Paths

- Main Obsidian vault: `OBSIDIAN_VAULT`; for this user, `{{OBSIDIAN_VAULT}}`
- Default work root: `WORK_ROOT`; for this user, `{{WORK_ROOT}}`
- Project cockpit root: `專案庫` inside the Obsidian vault
- Project cockpit note: `專案庫/<project-name>/專案工作流程.md` inside the Obsidian vault
- Optional personal assistant global layer: `ASSISTANT_ROOT`; for this user, `{{SYNC_ROOT}}`
- Optional personal assistant global entry skill: `$CODEX_HOME/skills/arry-assistant/SKILL.md`, or `~/.codex/skills/arry-assistant/SKILL.md` when `$CODEX_HOME` is not set
- Optional personal assistant setup document: ask the user for the path if no local setup document exists
- Global assistant skills: `ASSISTANT_ROOT/skills`
- Project-local assistant folders: `100_Todo/`, `200_Reference/`, and optional `000_Agent/skills/`, `000_Agent/memories/`
- Standard task folders: `100_Todo/drafts/`, `100_Todo/projects/`, `100_Todo/archive/`
- Standard reference folders: `200_Reference/writing-samples/`, `200_Reference/templates/`, `200_Reference/past-work/`

When adapting this skill for another user, replace `OBSIDIAN_VAULT`, `WORK_ROOT`, `ASSISTANT_ROOT`, and `ASSISTANT_NAME` before creating files. If the user does not want a personal assistant layer, skip assistant-specific folders and sections.

## Workflow

1. Inspect the current folder first. Report what already exists and what is missing.
2. Confirm missing facts before creating files:
   - project name
   - purpose
   - work folder
   - GitHub repo needed or not
   - public or private repo
   - GitHub Pages needed or not
   - Obsidian cockpit path
   - Firebase needed or not
   - Netlify needed or not
   - deployment target
3. For existing projects, only fill gaps. Do not overwrite existing `AGENTS.md`, `README.md`, `.gitignore`, Firebase config, or Git history.
4. Create or update the project root `AGENTS.md` with:
   - project name, purpose, work directory
   - GitHub repo or `未建立`
   - default branch
   - Obsidian vault absolute path
   - cockpit note vault-relative path
   - GitHub Pages status or `未啟用`
   - Firebase project id or `未使用`
   - Netlify site/team/deploy status or `未使用`
   - startup/shutdown/new-project rules
   - optional personal assistant integration: always bring in the shared global personal-assistant entry skill when the user uses Arry Assistant, plus the shared core layer, project-local `100_Todo/`, `200_Reference/`, and local `000_Agent/skills` when project-only skills are needed
   - safety rules: no secrets, no personal or sensitive data
5. Create or fill `README.md`, `.gitignore`, and Git repo if missing.
6. Create the project-local data layer if missing:
   - `100_Todo/drafts/`
   - `100_Todo/projects/`
   - `100_Todo/archive/`
   - `200_Reference/writing-samples/`
   - `200_Reference/past-work/`
   - `200_Reference/templates/`
   - `000_Agent/skills/` only when the project needs project-specific assistant skills or local workflows
   - `000_Agent/memories/` only when the project needs project-specific assistant memory separate from the global Arry core
7. Create the Obsidian cockpit note at `專案庫/<project-name>/專案工作流程.md` in the vault, not inside the project repo.
   - Include a deployment status block that records GitHub Pages, Firebase, Netlify, and the overall deployment target. Keep Netlify listed directly below Firebase.
   - If a personal assistant layer is used, include a short assistant integration section stating the model: shared global assistant core, project-local task/reference folders, and Obsidian as the knowledge/cockpit layer.
8. If GitHub is requested, use GitHub CLI or GitHub connector after auth verification. Public repo is preferred only when GitHub Pages is needed.
9. If Firebase is requested, keep `.firebaserc`, `firebase.json`, and rules scoped to the selected Firebase project. Never change Firebase project ID casually.
10. If Netlify is requested, verify Netlify CLI/MCP availability first, then confirm the Netlify team, site name or site ID, output folder, and production vs preview intent before creating or linking a site.
11. Finish by reporting:
   - work folder
   - Obsidian cockpit path
   - Git status
   - GitHub repo status
   - Firebase status
   - Netlify status
   - next action for the user

## Safety Rules

- Do not commit `.codex/`, `.claude/`, `.env`, API keys, tokens, passwords, admin credentials, personal data, or sensitive data.
- Do not store unnecessary personal or sensitive data in the repo.
- AGENTS.md stores fixed rules. Obsidian cockpit stores progress and next steps.
- A personal assistant layer, if used, has a two-layer model. Do not duplicate or move the shared assistant core layer.
- Every new project should get project-local `100_Todo/` and `200_Reference/` folders unless the user explicitly opts out.
- If the user asks for project-local assistant skills, create them under `000_Agent/skills/` and do not symlink that folder to `$CODEX_HOME/skills`.
- If the user asks for project-local assistant memory, create it under `000_Agent/memories/` and document how it syncs to the project cockpit.
- Any new skill must be portable: global skills sync to LazyPack and Obsidian global skill index; project skills stay as complete packages under project `000_Agent/skills/` and are recorded in the project cockpit.

# Startup Sync

Use this workflow at the beginning of a work session.

1. Read the project `AGENTS.md` if it exists.
2. Find and read the Obsidian cockpit note listed in `AGENTS.md`.
3. If the project references a personal assistant skill such as `arry-assistant`, read:
   - `$CODEX_HOME/skills/<assistant-skill>/SKILL.md`, or `~/.codex/skills/<assistant-skill>/SKILL.md` when `$CODEX_HOME` is not set
   - `ASSISTANT_ROOT/memories/MEMORY.md` when the project documents a global assistant root
4. Check Git status and current branch.
5. Check GitHub remote if present.
6. Check GitHub Pages if the project uses Pages or the cockpit says Pages is enabled.
7. Check Firebase project if the project uses Firebase.
8. Check Netlify if the project uses Netlify, has `.netlify/` or `netlify.toml`, or the cockpit says Netlify is enabled. Use read-only checks unless the user asks to link or deploy.
9. Summarize:
   - last known state
   - current local changes
   - GitHub Pages / Firebase / Netlify deployment status when relevant
   - likely next step
   - any blockers
   - personal assistant notes or preferences that apply to this project, if any

Do not automatically pull, commit, push, deploy, or rewrite rules unless the user asks.

# Shutdown Sync

Use this workflow at the end of a work session.

1. Review what changed during this session.
2. Read project `AGENTS.md` and the Obsidian cockpit note.
3. If the project uses a personal assistant layer such as Arry 助手, decide whether the session produced reusable cross-project memory:
   - reusable preference or repeated lesson -> append to `ASSISTANT_ROOT/memories/MEMORY.md` when that durable memory path is documented and approved
   - project-only progress -> keep in the project cockpit
4. Update the Obsidian cockpit with:
   - what was completed
   - current status
   - next steps
   - blockers or issues
5. Update `AGENTS.md` only if stable project rules, paths, or boundaries changed.
6. Check Git diff and status.
7. Stage only files related to this session.
8. Commit and push only when the user asked for it or clearly expects it.
9. Report:
   - Obsidian update status
   - personal assistant memory sync status, if used
   - AGENTS.md update status
   - Git status
   - any remaining uncommitted files

Never commit secrets, local `.codex/` data, or unrelated changes.
