# 11-外部工具整合工作流

## 目標

把外部工具整合流程轉成 Codex App 可用的全域 skill，讓 Codex 在遇到「整合、安裝、連線、驗證、選擇 API / CLI / MCP / connector 路線」時，有固定判斷流程。

這份文件只處理外部工具整合，不會套用到一般寫筆記、做簡報、整理文件、分析程式碼等普通 skill 呼叫。

- 已完成 `README.md` 的設定表。
- 已知道 Codex skills 位置：`{{CODEX_HOME}}/skills`。
- 已完成或讀過 `10-Codex版SkillCreator工作流.md`。
- 若使用 Obsidian 全域 skill 索引，已知道位置：

```text
{{OBSIDIAN_PROJECTS}}/{{SETUP_PROJECT_NAME}}/全域 Skills/全域 Skills 同步.md
```

## 固定結論

- skill 名稱：`tool-integration-workflow`
- 正式安裝位置：

```text
{{CODEX_HOME}}/skills/tool-integration-workflow/
```

- 懶人包內已附可直接複製的 skill 原始檔：

```text

{{SETUP_REPO}}/lazy-pack/skills/tool-integration-workflow/references/integration-route-checklist.md
```

- 外部工具整合路線優先順序：
  1. Codex connector / plugin
  2. 官方 CLI
  3. REST API + 本機 `.env`
  4. MCP
  5. 瀏覽器控制

下載本 repo 後，將整個 skill 資料夾複製到自己的 Codex skills 位置：

```bash
mkdir -p "{{CODEX_HOME}}/skills"
cp -R "{{SETUP_REPO}}/lazy-pack/skills/tool-integration-workflow" "{{CODEX_HOME}}/skills/"
```

如果你已經有同名 skill，先備份再覆蓋：


## 何時會觸發

使用者提出以下需求時應使用：

- 「幫我整合 Firecrawl / Notion / Slack / Linear」
- 「幫我安裝這個 API / CLI / MCP」
- 「確認 Codex 能不能使用這個工具」
- 「這個服務要走 CLI、API、MCP 還是 connector？」
- 「把某個外部服務接進我的助手或新專案工作流」

不應用在：

- 一般寫作或筆記整理。
- 一般簡報製作。
- 一般文件、試算表、圖片處理。
- 已經有明確 connector 可用且只是單次查詢的任務，除非需要驗證或設定整合。

## 工作流程

1. 先確認使用者真正要整合的服務與用途。
2. 檢查目前是否已有 Codex connector、plugin、CLI、API 設定或 MCP。
3. 用優先順序選最輕且穩定的路線：
   - Codex connector / plugin：適合 Gmail、Google Calendar、Google Drive、GitHub 等 Codex App 已支援工具。
   - 官方 CLI：適合 GitHub CLI、Firebase CLI、Heptabase CLI 等穩定命令列工具。
   - REST API + 本機 `.env`：適合需要精準控制且 API 穩定的服務。
   - MCP：只有 connector、CLI、API 不適合，或 MCP 明顯更好時才使用。
   - 瀏覽器控制：最後手段，適合 UI-only 或本機前端驗證。
4. 安裝前說明需要的權限與憑證，不自動加裝相鄰工具。
5. 安裝或設定後做一個低風險實測。
6. 把結果記錄到專案整合計畫、AI 能幹清單、工具清單或 Obsidian 駕駛艙。

## 安全規則

- 不把 API key、token、密碼、OAuth secret、app password、admin credential 寫進：
  - GitHub
  - repo 檔案
  - `AGENTS.md`
  - skill 檔案
  - README / docs
  - 公開或可能同步的 Obsidian 筆記
  - 截圖或聊天摘要
- 若使用者貼出 secret，不要原文重複；只說明應放在哪個本機安全位置。
- commit / push 前，先確認沒有 secret-bearing file 被納入。
- 需要服務權限時，優先選最小權限與 read-only。

## Firecrawl 範例

Firecrawl 適合用來抓取公開網頁內容、整理文章或研究資料。若已在本機跑通，不需要重新安裝，只要當次驗證。

驗證方式：

```text
請 Codex 使用 Firecrawl scrape https://example.com，確認回傳 Markdown 與 HTTP 200。
```

API key 原則：

- 只放在本機安全設定。
- 不上傳 GitHub。
- 不寫入 `AGENTS.md`、skills、README、docs 或 Obsidian 公開同步筆記。

## 踩坑修正

- 不要把 Claude Code 的 `~/.claude.json` 或 `~/.claude/skills` 安裝方式直接套到 Codex App。
- 不要因為某份教學列出很多 MCP 就全部安裝；只裝使用者確認需要的工具。
- Codex App 已有 connector 時，通常先用 connector，不必另裝 MCP。
- MCP / plugin / skill 設定完成後，通常要開新 Codex 對話或重啟 App。
- Firecrawl、Firebase、Notion 等 API key 不應出現在 Git diff 裡。
