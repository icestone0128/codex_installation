## 何時會觸發

使用者提出以下需求時應使用：

- 「幫我整合 Firecrawl / Notion / Slack / Linear」
- 「幫我安裝這個 API / CLI / MCP」
- 「確認 Codex 能不能使用這個工具」
- 「這個服務要走 CLI、API、MCP 還是 connector？」
- 「把某個外部服務接進我的助手或新專案工作流」
- 「先整理我的數位工具清單，不要安裝」
- 「先產生整合計畫，之後再做」

不應用在：

- 一般寫作或筆記整理。
- 一般簡報製作。
- 一般文件、試算表、圖片處理。
- 已經有明確 connector 可用且只是單次查詢的任務，除非需要驗證或設定整合。

## 工作流程

1. 先確認使用者真正要整合的服務與用途。
2. 檢查目前是否已有 Codex connector、plugin、CLI、API 設定或 MCP。
3. 先確認是「只產生計畫」、「稽核現況」或「實際安裝」，再用優先順序選最輕且穩定的路線：
   - Codex connector / plugin：適合 Gmail、Google Calendar、Google Drive、GitHub 等 Codex App 已支援工具。
   - 官方 CLI：適合 GitHub CLI、Firebase CLI、Heptabase CLI 等穩定命令列工具。
   - REST API + `~/.codex/secrets/<service>_api_key`：適合需要精準控制且 API 穩定的服務。
   - MCP：只有 connector、CLI、API 不適合，或 MCP 明顯更好時才使用。
   - 瀏覽器控制：最後手段，適合 UI-only 或本機前端驗證。
4. 安裝前說明需要的權限與憑證，不自動加裝相鄰工具。
5. 安裝或設定後做一個低風險實測。
6. 把結果記錄到專案整合計畫、AI 能幹清單、工具清單或 Obsidian 駕駛艙。

Codex 設定範圍固定為：

- 全域：`~/.codex/config.toml`
- 專案：`<project-root>/.codex/config.toml`
- 不直接 symlink 不相容的設定檔；依功能轉成 Codex TOML 後再驗證

## 安全規則

- 不把 API key、token、密碼、OAuth secret、app password、admin credential 寫進：
  - GitHub
  - repo 檔案
  - `AGENTS.md`
  - skill 檔案
  - README / docs
  - 公開或可能同步的 Obsidian 筆記
  - 截圖或聊天摘要
- 若使用者貼出 secret，不要原文重複；只說明應放在哪個本機安全位置。
- API key 與 token 預設放 `~/.codex/secrets/` 的 snake_case 檔名；環境變數只作執行期載入，不把 repo `.env` 當長期主版。
- commit / push 前，先確認沒有 secret-bearing file 被納入。
- 需要服務權限時，優先選最小權限與 read-only。

## Notion 範例

Notion 優先走 Codex Notion plugin / connector。只有 plugin 不能完成的特殊 API 工作，才再評估 REST API + 本機 `.env` 或 MCP。

安裝與驗證方式：

1. 啟用 Notion plugin / connector。
2. 連接自己的 Notion workspace。
3. 用只讀方式驗證，例如讀取目前連線使用者資訊，或搜尋使用者指定的測試頁面。
4. 若要建立或更新頁面，先取得使用者明確確認。
5. 若要寫入 database，先 fetch 目標 database / data source schema，確認正確 property 名稱後再寫入。

安全原則：

- 不手動保存 Notion token，除非 plugin / connector 無法滿足需求且使用者明確同意 API 路線。
- 不把 Notion token、workspace ID、page ID、database ID 或權限細節寫進 GitHub、repo、`AGENTS.md`、skills、README、docs 或公開同步筆記。
- 寫入 Notion 前要先說明目標位置與操作內容；不要自動建立、搬移、覆寫或刪除頁面。

# Tool Integration Workflow

Use this skill only for external tool integration work. Do not apply it to ordinary skill calls such as writing notes, making slides, editing documents, or analyzing code unless the task requires adding, verifying, or choosing an external service integration.

## Trigger Examples

- "幫我整合 Firecrawl / Notion / Slack / Linear"
- "幫我安裝這個 API / CLI / MCP"
- "確認 Codex 能不能使用這個工具"

## Core Rules

1. Confirm the user's actual use case before installing or configuring anything.
2. Prefer the lightest stable route that fits the task:
   - Codex connector or plugin when Codex App already provides a safe first-party workflow.
   - Official CLI when the service has a stable CLI and commands are enough.
   - REST API plus a local secret loaded from `~/.codex/secrets/<service>_api_key` when precise read/write control is needed.
   - MCP only when connector, CLI, or API are not suitable, or when MCP gives a clearly better agent experience.
   - Browser automation only as a last resort for UI-only workflows or local frontend verification.
3. For current tool choice, verify live docs or the currently available Codex tools when the route may have changed.
4. Install or configure only what the user confirmed. Do not add adjacent tools because they seem useful.
5. After setup, run one real, low-risk verification action.
6. Never write API keys, tokens, passwords, OAuth secrets, app passwords, admin credentials, or private keys into GitHub, repos, `AGENTS.md`, skills, public Obsidian notes, docs, README files, screenshots, or chat summaries.
7. If a global skill is added or changed as part of the integration, update the Obsidian global skill mirror note.
8. Keep Codex configuration scope explicit:
   - user-wide configuration: `~/.codex/config.toml`
   - project-specific configuration: `<project-root>/.codex/config.toml`
   - convert settings by intent and validate them; do not link or copy an incompatible configuration format into either path

## Mode Selection

- Planning-only mode: inventory current tools, recommend connector/CLI/API/MCP/browser routes, and produce a plan without installing or changing configuration.
- Implementation mode: execute only the confirmed items, one integration at a time, then verify each with a low-risk real action.
- Audit mode: inspect an existing integration, auth state, config scope, and failure layer before changing anything.

## Standard Workflow

1. Identify the target service and desired jobs-to-be-done.
2. Check what already exists:
   - Available Codex connectors/plugins/tools in the current session.
   - Local CLI install and auth state when relevant.
   - Existing project notes, skills, or integration plans.
3. Choose the route using the core rules and confirm whether the user wants planning-only, audit, or implementation.
4. Explain the selected route and any credentials needed before setup.
5. Execute only the confirmed setup.
6. Verify with a small real action:
   - Firecrawl: scrape one public URL.
   - GitHub: read repo/PR/issue status.
   - Gmail: read/search a safe mailbox query.
   - Calendar: read today's or a specified day's events.
   - Notion: read a test page or database after permission is granted.
7. Record the result in the relevant project plan or capability list.

Optional durable outputs, only when the user asks to save them and the project uses these layers:

- integration plan: `<project-root>/100_Todo/projects/YYYY-MM-DD-tool-integration-plan.md`
- confirmed capability inventory: `<project-root>/200_Reference/tool-capabilities.md`

## Secret Handling

- Store API keys and tokens under `~/.codex/secrets/` using readable snake_case file names, or use a provider-managed auth flow.
- Use environment variables only as runtime loaders; do not make a project `.env` the durable source of truth.
- Do not print full secret values.
- If a user pastes a secret, avoid repeating it; tell them where it should be stored safely.
- Before commit/push, check that no secret-bearing file is staged.

# Integration Route Checklist

This reference expands the `tool-integration-workflow` skill. Keep `SKILL.md` compact; use this file when a real external tool integration needs more structure.

## 1. Interview The Use Case

Ask only what is needed:

- What tool or service should be connected?
- What should Codex be able to do with it?
- Is read-only enough, or is write access required?
- Is this for one project, all future projects, or Arry assistant global use?
- Are credentials already available locally?

If the user only asks for analysis, do not configure anything until they approve execution.

## 2. Route Decision

Use this order:

1. Codex connector / plugin
   - Best when the connector already exists and handles auth safely.
   - Typical examples: Gmail, Google Calendar, Google Drive, GitHub.
2. Official CLI
   - Best when a stable CLI exists and the workflow is command-oriented.
   - Verify with a real command, not only `which`.
3. REST API + local environment
   - Best for precise read/write behavior and stable official APIs.
   - Store credentials outside tracked files.
4. MCP
   - Best when the service benefits from agent-native tools or no simpler route fits.
   - Keep MCP count small because every tool definition adds overhead.
5. Browser automation
   - Best for local frontend tests, UI-only services, or temporary verification.
   - Use logged-in browser automation only when the task requires the user's existing session.

Codex configuration scope:

- user-wide MCP/config: `~/.codex/config.toml`
- project-only MCP/config: `<project-root>/.codex/config.toml`
- do not share incompatible config formats by symlink; convert the intent into Codex TOML and verify the result

## 3. Setup Safety

- Do not put secrets in:
  - GitHub
  - repo files
  - `AGENTS.md`
  - skill files
  - Obsidian notes that may sync publicly
  - documentation or screenshots
- Durable API keys belong in `~/.codex/secrets/<service>_api_key`, not a repo `.env`.
- Do not widen filesystem or service permissions unless the user explicitly agrees.
- Prefer read-only permissions when the requested task does not require writing.
- For OAuth or destructive actions, make the action and scope clear before proceeding.

## 4. Verification Patterns

Every integration needs a real check:

- Firecrawl: scrape `https://example.com` or another public URL and confirm markdown plus HTTP 200.
- GitHub: read repo metadata or current PR status.
- Gmail: run a safe search or summarize a small thread.
- Calendar: read events for a specific date.
- Google Drive: read a non-sensitive file the user selected.
- Notion: read a test page or database after the integration is invited.
- Heptabase: list or read a safe card through the CLI.

Verification should not create, send, delete, publish, or modify data unless the user explicitly asked for that.

## 5. Documentation After Setup

Record:

- Actual route used.
- Whether it was newly installed, already installed, or only verified.
- Where the safe configuration lives, without secret values.
- The verification action and result.
- Any limitations, quota notes, or permission boundaries.

Common project destinations:

- `100_Todo/projects/<date>-tool-integration-plan.md`
- `200_Reference/tool-capabilities.md`
- Obsidian project cockpit when the integration changes project workflow.
